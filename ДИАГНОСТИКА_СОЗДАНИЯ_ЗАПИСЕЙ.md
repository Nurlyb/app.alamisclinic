# Диагностика проблемы создания записей

## Проблема
Регистратор не может создать запись - возникает ошибка "недостаточно прав".

## Добавленные инструменты диагностики

### 1. Подробное логирование
Добавлено детальное логирование в:
- `lib/auth/middleware.ts` - логи аутентификации и авторизации
- `app/api/appointments/route.ts` - логи создания записи
- `lib/auth/rbac.ts` - логи проверки прав

### 2. Тестовый endpoint
Создан `/api/test/appointment-creation` для проверки прав без создания записи.

### 3. HTML тесты
- `test-permissions-simple.html` - простой тест прав и создания записи
- `test-appointment-creation.html` - детальный тест API

### 4. Панель отладки
В правом нижнем углу страницы расписания:
- 🧪 Прямой тест прав
- 🔍 Проверка сервера и БД
- 🔑 Обновление токена

## Пошаговая диагностика

### Шаг 1: Проверьте логи сервера
1. Откройте консоль сервера
2. Попробуйте создать запись
3. Найдите логи с префиксами:
   - `🔐 withAuth:` - вход в middleware
   - `✅ Authenticated user:` - успешная аутентификация
   - `🔍 Checking permissions:` - проверка прав
   - `❌ Authorization failed` - ошибка авторизации
   - `📝 POST /api/appointments` - начало создания записи

### Шаг 2: Используйте панель отладки
1. Откройте страницу расписания
2. Нажмите кнопку 🧪 для прямого теста прав
3. Проверьте результат в панели
4. Если показывает НЕТ - нажмите 🔑 для обновления токена

### Шаг 3: Используйте HTML тесты
1. Откройте `test-permissions-simple.html` в браузере
2. Получите access token (из DevTools → Application → Local Storage)
3. Нажмите "Тестировать права"
4. Проверьте результат

### Шаг 4: Проверьте данные в БД
Выполните SQL запрос:
```sql
SELECT id, name, role, isActive FROM users WHERE role = 'RECEPTIONIST';
```

### Шаг 5: Проверьте JWT токен
1. Скопируйте access token из localStorage
2. Вставьте на jwt.io
3. Проверьте поле `role` в payload

## Возможные причины и решения

### 1. Устаревший токен
**Симптомы:** Роль в токене не соответствует роли в БД
**Решение:** Нажмите 🔑 в панели отладки или перелогиньтесь

### 2. Неактивный пользователь
**Симптомы:** `isActive: false` в БД
**Решение:** Активируйте пользователя в БД:
```sql
UPDATE users SET "isActive" = true WHERE role = 'RECEPTIONIST';
```

### 3. Проблема с правами в коде
**Симптомы:** Логи показывают `❌ Authorization failed`
**Решение:** Проверьте `lib/auth/rbac.ts` - роль RECEPTIONIST должна иметь `appointments:create`

### 4. Проблема с валидацией данных
**Симптомы:** Ошибка валидации в логах
**Решение:** Проверьте корректность данных формы (ID пациента, доктора, услуги)

### 5. Проблема с базой данных
**Симптомы:** Ошибка при создании записи в БД
**Решение:** Проверьте подключение к БД и корректность данных
