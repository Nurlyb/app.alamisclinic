# Исправление доступа ассистента к расписанию

## Проблема
Ассистенты получали ошибку 403 (Forbidden) при попытке доступа к странице расписания и API endpoints.

## Причина
В файле `app/api/users/[id]/route.ts` была слишком строгая проверка доступа:
```typescript
if (user.userId !== id && user.role !== 'OWNER') {
  return errorResponse('Нет доступа', 'FORBIDDEN', 403);
}
```

Эта проверка разрешала доступ только:
- К своему собственному профилю
- Владельцу (OWNER) для доступа к любому профилю

Но ассистентам нужно было получить свой профиль для загрузки `assistingDoctorId`, чтобы увидеть записи своего доктора.

## Решение
Обновлена логика проверки доступа в `app/api/users/[id]/route.ts`:

```typescript
const canAccess = 
  user.userId === id || // Свой профиль
  user.role === 'OWNER' || // Владелец видит всех
  (user.role === 'ASSISTANT' && user.userId === id) || // Ассистент видит свой профиль
  (user.role === 'DOCTOR' && user.userId === id); // Доктор видит свой профиль

if (!canAccess) {
  return errorResponse('Нет доступа', 'FORBIDDEN', 403);
}
```

### 2. Фильтрация записей в API
В `app/api/appointments/route.ts` добавлена логика для ассистентов:
```typescript
else if (user.role === 'ASSISTANT') {
  const assistant = await prisma.user.findUnique({
    where: { id: user.userId },
    select: { assistingDoctorId: true },
  });
  
  if (assistant?.assistingDoctorId) {
    where.doctorId = assistant.assistingDoctorId;
  } else {
    return successResponse({ data: [] });
  }
}
```

### 3. Фильтрация на фронтенде
В `app/schedule/page.tsx` добавлена фильтрация:
```typescript
// Ассистент видит записи своего доктора
if (user?.role === 'ASSISTANT' && assistantInfo?.assistingDoctorId) {
  return apt.doctorId === assistantInfo.assistingDoctorId;
}
```
